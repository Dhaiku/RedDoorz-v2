Document structure · Data entry · Data manipulation · Transaction processing

Screen
Open the Firebase Console → Firestore Database. Navigate through the collections one by one.
Narration
"Let's now look at exactly how data is stored in Google Cloud Firestore. Unlike a relational SQL database — which stores data in rows and tables with foreign key joins — Firestore is document-based. Each record is a JSON-like document inside a named collection. Related data is linked by storing IDs — like hotelId or custId — inside the document. There are no joins. The PHP application resolves those relationships using individual document lookups."

Narration
"When a customer registers, two documents are written in sequence. First, an accounts document is created to store login credentials — the password is hashed with PHP's password_hash() using bcrypt before being stored. Then, a customers document is created with the profile information, storing the account ID as a reference. This separation follows the single-responsibility principle — the accounts collection owns authentication, the customers collection owns identity."

Auto-Increment IDs — The Counters Collection
Screen
Open the counters collection. Show documents for bookings, accounts, hotels — each with a single next field.
Narration
"Since Firestore doesn't have an equivalent to SQL's AUTO_INCREMENT, this application implements its own using a counters collection. Each document in counters is named after a collection and has a single field — next — which holds the next integer ID to assign. Before creating any new document, the system reads this counter, increments it, and uses the current value as the new document's ID. This preserves the integer ID behavior from the original MySQL design."

1 — The Firestore REST Client (config/firestore.php)
Screen
Open config/firestore.php in VS Code. Scroll to show the public API functions: fs_get(), fs_query(), fs_insert(), fs_update(), fs_delete().
Narration
"This file is the heart of the application. Because PHP has no official Firestore SDK, this custom client communicates with Firestore's REST API directly over HTTPS using cURL. Every page that reads or writes data calls these five functions. The client handles OAuth2 token acquisition, JSON encoding and decoding of Firestore's wire format, and a two-level cache to minimize network calls."
Key Methods — config/firestore.php
// Read one document by integer ID
fs_get('hotels', 42);           // → array|null

// Multi-condition query with optional sort & limit
fs_query('bookings', [
    ['custId', '=', 7],
    ['status', '=', 'confirmed'],
], [['createdAt', 'DESC']], 10);

// Insert — auto-increments ID from counters collection
$id = fs_insert('bookings', ['custId' => 7, 'status' => 'pending', ...]);

// Partial update — only named fields are patched
fs_update('bookings', $id, ['status' => 'confirmed']);

// Delete a document
fs_delete('reviews', $reviewId);
2 — Authentication (auth/login.php)
Screen
Open auth/login.php. Show the POST handler — the Firestore query, password verification, and session setup.
Narration
"Authentication is entirely session-based. When a user submits the login form, the code queries the accounts collection for a document matching their email. It then runs PHP's password_verify() to check the submitted password against the stored bcrypt hash. On success, the role is read from the document and the appropriate session variables are set — account_id and role. Every protected page begins with a session check: if these variables are missing or the role doesn't match, the user is redirected to login."
Auth logic — auth/login.php
$acct = fs_find('accounts', [['email', '=', $email]]);

if (!$acct || !password_verify($password, $acct['password'])) {
    // Invalid credentials
    header("Location: login.php?error=1"); exit();
}

$_SESSION['account_id'] = $acct['id'];
$_SESSION['role']       = $acct['role'];

// Role-based redirect
match($acct['role']) {
    'admin'       => header("Location: /admin/dashboard.php"),
    'hotel_owner' => header("Location: /owner/dashboard.php"),
    'customer'    => header("Location: /customer/dashboard.php"),
};
3 — Booking & Payment Flow (hotels/book.php)
Screen
Open hotels/book.php. Show the POST handler — validation, booking insert, payment insert, earnings calculation.
Narration
"The booking creation code first validates all required fields — dates must not be in the past, check-out must be after check-in, and guest count must not exceed room capacity. Earnings split — hotels/payment.php (POST handler)
$total      = $booking['totalPrice'];
$ownerShare = round($total * 0.85, 2);
$platFee    = round($total * 0.15, 2);

// 1. Confirm the booking
fs_update('bookings', $bookId, ['status' => 'confirmed']);

// 2. Record payment
fs_insert('payments', [
    'bookId' => $bookId, 'amount' => $total,
    'method' => $method, 'status' => 'paid',

// 3. Create earnings record

4 — Performance: Two-Level Cache
Screen
Open config/firestore.php. Scroll to the cache layer section — show _fs_cache_get() and _fs_cache_set().
Narration
"Every Firestore query passes through a two-level cache before hitting the network. The first level is a static PHP array — it lives in memory for the current request, giving zero-millisecond lookups when the same data is needed multiple times on the same page. The second level is a disk cache — JSON files stored in the cache/ directory. These survive across page loads, meaning a collection like hotels, which rarely changes, is served from disk in about one millisecond instead of making a round-trip to Google's servers.

Cache TTLs are set per-collection — ten minutes for hotels and accounts, five minutes for rooms, two minutes for bookings. Every write operation immediately busts the cache for that collection, so data is always consistent after any change."
